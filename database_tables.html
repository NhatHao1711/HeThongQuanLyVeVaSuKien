3.3.2. Thiết kế chi tiết các bảng dữ liệu

Dưới đây là đặc tả cấu trúc vật lý chi tiết của các bảng trong cơ sở dữ liệu hệ thống quản lý sự kiện TRIVENT:

1. Bảng users (Quản lý thông tin tài khoản)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã định danh người dùng tự tăng
university_id BIGINT FOREIGN KEY REFERENCES universities(id), NULL Mã trường đại học của sinh viên
email VARCHAR(150) NOT NULL, UNIQUE Địa chỉ email đăng nhập (Xác thực duy nhất)
full_name VARCHAR(150) NOT NULL Họ và tên hiển thị trên hệ thống
password_hash VARCHAR(255) NOT NULL Chuỗi mật khẩu băm bảo mật bằng Bcrypt
phone VARCHAR(20) NULL Số điện thoại liên lạc
avatar_url VARCHAR(500) NULL Đường dẫn ảnh đại diện người dùng
role VARCHAR(20) NOT NULL, DEFAULT 'ROLE_USER' Vai trò (ROLE_USER, ROLE_ORGANIZER, ROLE_ADMIN)
is_verified BIT(1) DEFAULT 0 Trạng thái xác thực email trường học (0: Chưa, 1: Rồi)
interests_tags JSON NULL Mảng chứa các thẻ sở thích phục vụ ghép đôi
balance DECIMAL(12,2) DEFAULT 0.00 Số dư ví tài khoản của người dùng (BTC/Admin)
holding_balance DECIMAL(12,2) DEFAULT 0.00 Số dư tạm giữ (doanh thu sự kiện chờ đối soát)
kyc_status VARCHAR(20) NULL Trạng thái định danh câu lạc bộ hoạt động
created_at DATETIME NOT NULL Thời điểm đăng ký tài khoản
updated_at DATETIME NULL Thời điểm cập nhật thông tin tài khoản gần nhất

2. Bảng events (Quản lý các chương trình / sự kiện ngoại khóa)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã định danh sự kiện tự tăng
organizer_id BIGINT FOREIGN KEY REFERENCES users(id) Mã tài khoản của Ban Tổ Chức tạo sự kiện
category_id BIGINT FOREIGN KEY REFERENCES event_categories(id) Thể loại/Danh mục của sự kiện
title VARCHAR(300) NOT NULL Tiêu đề của chương trình
description TEXT NULL Mô tả chi tiết nội dung sự kiện
location VARCHAR(500) NULL Địa điểm tổ chức sự kiện
image_url VARCHAR(500) NULL Ảnh bìa/Banner truyền thông
status VARCHAR(20) NOT NULL, DEFAULT 'DRAFT' Trạng thái duyệt (DRAFT, PENDING, PUBLISHED, CLOSED)
is_featured BIT(1) DEFAULT 0 Đánh dấu sự kiện nổi bật hiển thị ở Banner chính
reject_reason VARCHAR(500) NULL Lý do từ chối kiểm duyệt từ phía Nhà trường
views BIGINT DEFAULT 0 Số lượt xem trang chi tiết sự kiện
start_time DATETIME NOT NULL Thời gian bắt đầu diễn ra sự kiện
end_time DATETIME NOT NULL Thời gian kết thúc sự kiện
created_at DATETIME NOT NULL Thời điểm tạo đề xuất sự kiện

3. Bảng ticket_types (Quản lý các hạng đăng ký / vé sự kiện)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã hạng đăng ký tự sinh
event_id BIGINT FOREIGN KEY REFERENCES events(id) Thuộc về sự kiện nào
name VARCHAR(100) NOT NULL Tên hạng vé (Ví dụ: Vé VIP, Vé Thường, Guest)
price DECIMAL(12,2) NOT NULL Giá/Phí tham dự (0.00 nếu miễn phí)
total_quantity INT NOT NULL Số lượng suất đăng ký giới hạn tối đa
available_quantity INT NOT NULL Số lượng suất còn trống có thể đăng ký
event_date DATE NULL Ngày diễn ra (áp dụng cho sự kiện kéo dài nhiều ngày)

4. Bảng seats (Quản lý chi tiết ghế ngồi trong hội trường)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã định danh vị trí ghế
ticket_type_id BIGINT FOREIGN KEY REFERENCES ticket_types(id) Thuộc phân loại hạng vé nào
name VARCHAR(50) NOT NULL Tên vị trí ghế ngồi (Ví dụ: A01, B12...)
status VARCHAR(20) NOT NULL, DEFAULT 'AVAILABLE' Trạng thái ghế (AVAILABLE, LOCKED, BOOKED)
created_at DATETIME NOT NULL Thời điểm tạo ghế

5. Bảng user_tickets (Quản lý Thẻ tham dự điện tử và Check-in)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã định danh thẻ tham dự tự sinh
order_id BIGINT FOREIGN KEY REFERENCES orders(id) Thuộc đơn đặt chỗ nào
ticket_type_id BIGINT FOREIGN KEY REFERENCES ticket_types(id) Thuộc loại vé nào
user_id BIGINT FOREIGN KEY REFERENCES users(id) Sinh viên sở hữu thẻ mời
seat_id BIGINT FOREIGN KEY REFERENCES seats(id), NULL Vị trí ghế ngồi được gán (nếu có)
qr_token VARCHAR(500) UNIQUE Mã bảo mật mã hóa AES dùng quét QR
checkin_status VARCHAR(10) NOT NULL, DEFAULT 'UNUSED' Trạng thái điểm danh sinh viên (UNUSED, USED)
checkin_time DATETIME NULL Thời điểm sinh viên check-in vào cửa thực tế
created_at DATETIME NOT NULL Thời điểm xuất thẻ tham dự

6. Bảng orders (Quản lý các giao dịch / Đơn đăng ký)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã đơn hàng tự sinh
user_id BIGINT FOREIGN KEY REFERENCES users(id) Tài khoản sinh viên thực hiện đăng ký
total_amount DECIMAL(12,2) NOT NULL Tổng tiền đơn hàng
discount_amount DECIMAL(12,2) DEFAULT 0.00 Số tiền được giảm giá
voucher_code VARCHAR(50) NULL Mã giảm giá đã áp dụng
transaction_ref VARCHAR(100) NULL Mã tham chiếu đối soát của PayOS / VNPay
payment_method VARCHAR(20) NULL Phương thức thanh toán (VNPAY, PAYOS, BALANCE)
payment_status VARCHAR(20) NOT NULL, DEFAULT 'PENDING' Trạng thái thanh toán (PENDING, PAID, FAILED)
version BIGINT NOT NULL Biến phiên phục vụ khóa lạc quan (Optimistic Lock)
created_at DATETIME NOT NULL Thời điểm khởi tạo đơn hàng

7. Bảng event_buddies (Quản lý kết nối Bạn đồng hành)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã định danh ghép đôi tự tăng
event_id BIGINT FOREIGN KEY REFERENCES events(id) Đăng ký đi chung tại sự kiện nào
sender_id BIGINT FOREIGN KEY REFERENCES users(id) Sinh viên gửi yêu cầu ghép đôi
receiver_id BIGINT FOREIGN KEY REFERENCES users(id) Sinh viên nhận yêu cầu ghép đôi
status VARCHAR(20) NOT NULL, DEFAULT 'PENDING' Trạng thái yêu cầu (PENDING, ACCEPTED)
created_at DATETIME NOT NULL Thời điểm gửi yêu cầu kết nối

8. Bảng universities (Quản lý danh mục các trường đại học)

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết
id BIGINT PRIMARY KEY, AUTO_INCREMENT Mã trường tự tăng

Tên cột Kiểu dữ liệu Ràng buộc Mô tả chi tiết

3.3.3. Chỉ mục và tối ưu hóa truy vấn

Đối với hệ thống quản lý sự kiện dành cho sinh viên, việc đảm bảo tốc độ phản hồi nhanh dưới lượng truy cập đồng thời lớn là vô cùng quan trọng (đặc biệt là khi mở cổng đăng ký sự kiện lớn hoặc check-in hàng loạt tại cửa hội trường). Hệ thống áp dụng hai phương pháp tối ưu hóa dữ liệu cốt lõi sau:

1. Thiết lập các chỉ mục vật lý (Database Indexes)

Chỉ mục được xây dựng trên các cột thường xuyên xuất hiện trong mệnh đề WHERE, JOIN, ORDER BY nhằm giảm thiểu tối đa tài nguyên quét đĩa (Disk I/O):

Tên chỉ mục Bảng & Cột áp dụng Loại chỉ mục Mục đích tối ưu hóa
idx_users_email users (email) UNIQUE INDEX Tăng tốc truy vấn đối sánh tài khoản sinh viên/BTC khi đăng nhập và kiểm tra bảo mật token JWT.
idx_tickets_qr_token user_tickets (qr_token) UNIQUE INDEX Tăng tốc truy vấn tìm kiếm vé mời khi quét mã QR check-in tại cửa sự kiện, giúp thời gian phản hồi soát vé duy trì dưới 5ms.
idx_seats_type_status seats (ticket_type_id, status) COMPOSITE INDEX Tối ưu hóa tốc độ tải sơ đồ ghế trống hội trường. Hệ thống chỉ cần lọc nhanh các ghế có trạng thái 'AVAILABLE' của hạng vé mà không phải quét toàn bảng.
idx_events_status_time events (status, start_time) COMPOSITE INDEX Tối ưu hóa các câu lệnh hiển thị danh sách sự kiện đang hoạt động (PUBLISHED) và sắp xếp theo trình tự thời gian diễn ra trên trang chủ.
idx_buddies_event_status event_buddies (event_id, status) INDEX Tăng tốc độ truy vấn gợi ý và hiển thị danh sách ghép đôi đi chung sự kiện của sinh viên.
idx_ledger_agency ledger_entries (agency_id) INDEX Tối ưu hóa câu lệnh tính toán doanh thu lũy kế và lịch sử ví tiền của các câu lạc bộ (BTC).

2. Tối ưu hóa truy vấn ở tầng ứng dụng (JPA/ORM Optimization)

  • Giải quyết triệt để lỗi N+1 Query: Sử dụng cơ chế nạp dữ liệu sớm thông qua mệnh đề JOIN FETCH hoặc annotation @EntityGraph trong Spring Data JPA. Thay vì sinh ra hàng chục câu lệnh SELECT con để tải thông tin Ban tổ chức (User) và Danh mục (Category) của sự kiện, hệ thống gộp tất cả vào một câu SELECT duy nhất bằng lệnh LEFT JOIN FETCH.
  • Cơ chế khóa lạc quan (Optimistic Locking): Sử dụng cột version trong bảng orders thông qua annotation @Version của JPA. Khi có nhiều luồng tranh chấp đăng ký suất vé cuối cùng, hệ thống tự động kiểm tra số phiên bản của bản ghi; nếu phát hiện xung đột ghi đè, hệ thống sẽ chặn giao dịch sau và yêu cầu thử lại, bảo đảm tính toàn vẹn dữ liệu mà không cần khóa cứng bảng làm chậm hệ thống.
  • Cơ chế phân trang dữ liệu (Database Pagination): Áp dụng đối tượng Pageable ở mức Repository đối với tất cả API tải danh sách lịch sử đăng ký hoặc danh mục sự kiện lớn. Cơ chế này giới hạn số lượng dòng kết quả truyền về (OFFSET và LIMIT ở mức SQL), tránh tình trạng tràn bộ nhớ Heap của JVM khi dữ liệu phình to theo thời gian.
